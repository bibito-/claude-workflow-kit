# claude-workflow-kit

このリポジトリは **core（スタック非依存のワークフロー）の正**。各利用プロジェクトはここから pull して `.claude/` を同期する。

配布物の設計・実装は**このリポジトリで行う**。利用プロジェクト側で書いて push で還流させる経路（`/workflow-kit-push`）も残しているが、それは「実作業中に気づいた修正」のための副次経路であり、主戦場はここ。

## ディレクトリの役割

| パス | 役割 | 配布 |
|---|---|---|
| `.claude/` | **配布ペイロード**。利用プロジェクトの `.claude/` にそのまま入る | される（`.claude/manifests/workflow-kit-files.txt` に載っているものだけ） |
| `template/` | 新規プロジェクト作成時に一度だけ取り込む骨格。`{{PLACEHOLDER}}` 入り | 同期しない（scaffold で一度きり） |
| `docs/` | このリポジトリ自身の設計記録（spec・設計判断の永続アーカイブ） | されない |
| `scripts/` | scaffold スクリプト。kit クローンから直接実行する | されない |
| `.github/` | このリポジトリの CI | されない |

`.claude/settings.json` は配布対象外（利用プロジェクトごとに permissions・MCP 設定が異なるため）。ここに置いてあるのは**このリポジトリで作業するときの設定**。

## 絶対に守ること

### 配布ペイロードに特定の層の語彙を書かない

`.claude/` 配下（= マニフェスト掲載ファイル）に、特定のフレームワーク・BaaS・ランタイム・インフラ提供者・利用プロジェクトの名前を書いてはならない。core は**どのスタックのプロジェクトにも配られる**ため、書いた瞬間に配布先の大半で意味をなさなくなる。

CI（`kit-push-guard`）が PR の追加行を grep して弾く。検査対象はマニフェスト掲載ファイルのみで、`docs/` や `README.md` は対象外（設計記録が経緯として層の名前に触れるのは正当なため）。

層固有の語彙が必要になる場合は、**呼び出し側（スキル）が引数として渡す**設計にする。詳細は [.claude/rules/agent-definition-guide.md](.claude/rules/agent-definition-guide.md)。

### main への直 push は禁止

main は ruleset `protect-main`（bypass 空・PR 必須）で保護されている。変更はブランチ → PR → CI green → **マージはユーザーが実行する**。Claude はマージしない。

事故の最終的な検知手段は「他人の変更を消す hunk が diff に見える」ことであり、自動マージするとその diff を誰も見ないまま通る。

### push 前にレビューを受ける

`git push` は `kit-push-review-agent` の clean verdict が無いとフックにブロックされる（`.claude/hooks/guard-kit-push-verdict.cjs`）。verdict は差分の digest に束縛されており、レビュー後に1文字でも変えれば無効になる。

このリポジトリ内で書いたものにも効く。「利用プロジェクトからコピーしていないから安全」ではない — ここで書いても層固有の語彙を core に混ぜる事故は起こりうるし、CI の grep は単純一致しか見ない。

**審査が走るのは変更が配布物に触れたときだけ。** マニフェスト掲載ファイル・`template/`・`scripts/` と、審査スコープの宣言ファイル自身。それ以外（`README.md`・この `CLAUDE.md`・`docs/`・`.github/`・`.githooks/`）は外へ出ないため一切審査しない。審査1回はエージェント起動ぶんの実費であり、守るものが無い push には払わない。詳細は [docs/kit-push-gate.md](docs/kit-push-gate.md)。

## ガードの強度は対象の成熟度に比例させる

core はまだ流動的で、改修は続く。固まっていないものに固まった前提の錠前をかけない。

- 配布されるフックに方針をハードコードしない（変えるたびに全 kit が pull を強いられる）
- 試行錯誤が前提の時期に、再審査を誘発する束縛（digest 固定など）を強めない
- 構造が動く時期に、構造変更へ税をかける CI 強制を足さない

強制は安定してから。流動的な間は advisory で足りる。設計を「正典」として扱い、「聖典」にしない。

## コミット規約

- 修正・実装は意味のある単位ごとに独立してコミットする。複数の指摘をまとめて1コミットにしない
- `git add -A` / `git add .` は使わない。対象ファイルを明示的に列挙する
- コミットメッセージの**トレーラ行の間に空行を入れない**。git は最後の段落だけをトレーラブロックとして解釈するため、空行で分けると先頭のトレーラが認識されず CI が落ちる

## 回答スタイル

挨拶・前置き・段階報告・絵文字は禁止。結論ファースト。指摘すべきことは率直に指摘する。

応答は結論を端的に報告してから、必要な場合のみ補足説明を続ける。変更内容をコミットハッシュを主語にして語らず、「何をしたか」を平易な言葉で説明する。

## 絶対にやってはいけないこと

- 環境変数の値を出力しない
- 認証情報をハードコードしない
- 個人情報（メールアドレス・名前・電話）をログに出力しない
- シークレット・認証情報の登録操作は必ずユーザー自身が行う。コマンド例を提示するにとどめる
- `.claude/settings.json` の変更は自由だが、コミット・push は都度ユーザーの承認を得る
