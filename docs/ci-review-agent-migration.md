# 混入審査を CI へ移す構想

最終更新: 2026-07-12
ステータス: **未実装（構想・未解決問題あり）**

> 実装の合意はまだ無い。現行機構の非対称性を記録し、移行案とその障害を残すためのもの。

## 発端

「core にフレームワーク名を書いたら弾く」ガードの実体を追ったところ、**強い判定は迂回可能な経路にあり、無条件に走る経路の判定が一番弱い**という配置になっていることが分かった。

## 現状の配置

| | 縛る対象 | 判定 |
|---|---|---|
| `kit-push-review-agent` + `guard-kit-push-verdict.cjs` | **Claude の push のみ** | 文意レベル。`universality_criterion` を引数で受け取り、「手順文・動作依存の位置への固有名混入」「層を超えた前提」を見る |
| `kit-push-guard.yml` の禁止語 grep | **すべての PR（無条件）** | `FORBIDDEN: 'wrangler\|cloudflare\|supabase\|\bhono\b'` の固定4語 |

### フックは Claude しか縛らない

`guard-kit-push-verdict.cjs` は `PreToolUse` / `matcher: Bash` で登録されている（`.claude/manifests/hook-registrations.json`）。**Claude が Bash ツールで `git push` を叩くときにしか発火しない。**

発火しない経路:

- 人間が端末で直接 `git push` する
- GitHub の Web UI でファイルを編集して PR を立てる
- Claude 以外のツール・別のエージェント・別マシンから push する
- 利用プロジェクトが `merge-hook-registrations` を踏んでおらず、`settings.json` にフックが登録されていない（`settings.json` は配布対象外なので、pull 時のマージを経ていなければフックは存在しない）

これらは例外的な迂回ではなく、フックの設計上の射程外。フックは**リポジトリを縛るガードではなく、Claude を縛るガード**。

一方 main は PR 必須なので、上記のどの経路でも PR は必ず立ち、CI は必ず走る。**CI は Claude の協力に依存しない唯一の無条件ガード。**

### その唯一の無条件ガードが 4 語の grep である

grep が「境界に属する情報を弾く」ものになっていない点が 2 つある。どちらも「review agent が抜けた／見落としたときに CI が拾えるか」という話であり、review agent が正常に働く限り顕在化しない。だが CI の存在意義はまさにその「抜けたとき」を拾うことにある。

1. **別スタックの利用プロジェクトには効かない。** 禁止語は「いまこの kit を使っている人のスタック名」のハードコードでしかない。異なるスタックのプロジェクトから push されれば、そのスタックの語は素通りする。利用プロジェクトが増えるたびに禁止語を足す必要があり、これは「core が全利用プロジェクトの層を知っている」ことを要求する。**core をスタック非依存に保つという目的と原理的にねじれている。**

2. **フレームワーク名を 1 語も書かずに境界を侵犯できる。** ディレクトリ構成の前提（「`src/routes/` 配下を読む」）、手順の前提（「デプロイ前に必ず型チェックを通す」）、環境変数名、ビルドコマンド名。すべて展開先の境界に属する情報だが、grep には原理的に見えない。review agent の判定観点はここを見ているが、grep は見られない。

したがって CI は**境界検査ではなく、過去に実際に混入した語の再発防止ネット**として理解するのが正確。**CI が green だったことは、境界を越えていないことを何も意味しない。**

## 構想: 審査を CI 側へ移す

review agent を「配布物として展開先に置き、ローカルフックで強制する」のをやめ、**core リポジトリの GitHub Actions で PR に対して走らせる**。

### 消える足場

現行機構の大半は「ローカルフックを信頼できるゲートに仕立てる」ためだけに存在している。ゲートが CI にあるなら不要になる。

- `guard-kit-push-verdict.cjs`（digest 束縛して push をブロックする機構）
- `guard-kit-verdict-write.cjs`（verdict を書けるのは agent だけ、という書き込み制限）
- `.claude/steering/reviews/` の verdict ファイルと audit trail
- `hook-registrations.json` の該当 2 エントリと、その配布・マージ経路
- push スキルの verdict 取り回し
- 配布ペイロードから `kit-push-review-agent.md` 自体

verdict を偽造される心配も、digest で差分に束縛する必要も、書き手を制限する必要もない。**CI は Claude の協力なしに無条件で走るため、Claude が verdict を握りつぶす余地が構造的に無い。** いま重い機構でエミュレートしているものが、環境の性質としてタダで手に入る。

## 未解決問題

### 1. `universality_criterion` の出所

現行はこれを「呼び出し側スキル（＝利用プロジェクト側）が引数で持つ」と定義している。core は層を知らない、という原則の帰結。

だが CI は core リポジトリで走る。core が基準を持てない以上、**押す側が PR に申告して持ち込む**しかない。

案: push スキルがコミットトレーラか PR body に `universality_criterion` / `exclusion_list` を載せ、CI の review agent がそれを読んで判定する。既存の `Workflow-Kit-Base:` トレーラとまったく同じ形になり、agent 定義の思想（agent は層に無知、具体値は呼び出し側が管理）がそのまま CI に移る。

信頼モデルは劣化しない。現行も基準文は呼び出し側が渡しており、「緩い基準を渡せば緩く通る」性質は同じ。変わるのは**判定が走ることを保証できるかどうか**だけで、そこが改善点そのもの。

**未確定:** トレーラに載せるのか PR body に載せるのか。トレーラは 1 行に収まらない基準文を扱いにくい。

### 2. LLM 判定を required status check にする非決定性

CI で LLM を回すと判定が非決定的になる。それを main のマージ条件（required check）にすると、**フレーキーな required check** ができあがる。`kit-push-guard-ci.md` で既に「CI がバグって落ち続けると kit に対して誰も何も反映できなくなる」ことを了解済みの帰結として受け入れているが、確定的な grep で落ち続けるのと、LLM が気まぐれで落ちるのとでは性質が違う。

案: required にせず、**CI の review agent は PR にレビューコメントを付けるだけ**にする。`CLAUDE.md` の「マージはユーザーが実行する / 自動マージすると事故を示す diff を誰も見ないまま通る」という既存原則と真正面から整合する。ゲートの本質は「人間が見る前に判定が必ず提示されていること」であって、機械が自動で止めることではない。

**未確定:** advisory に落とすと、ユーザーがコメントを読まずにマージできる。それを許容するか、`contaminated` のときだけ required 相当に落とす二段構えにするか。

### 3. 禁止語 grep の扱い

LLM を呼ぶ前の安い事前フィルタとして残す価値はある（確実な混入を即座に落とす）。ただし残す場合も、**「あれが通ったから安全」という意味づけは捨てる**。現状の禁止語リストが「境界の定義」であるかのように読める書き方は改める。

### 4. API キーと課金

CI で LLM を回すには Anthropic API キーを GitHub Actions secret に置く必要がある。core リポジトリ 1 箇所で済むが、シークレット登録操作はユーザー自身が行う。PR ごとに課金が発生する点も込みで判断する。

## 別テーマ: 審査の摩擦

CI 移行とは別に、**現行フックの摩擦**が問題として挙がった。フックはマニフェストのスコープを持たず、kit クローンからの全 push に verdict を要求する。`docs/` に markdown を 1 本置くだけ、`README` の誤字を 1 文字直すだけでも agent 起動（数十秒・数万トークン）が要る。`CLAUDE.md` が「主戦場は kit リポジトリでの直接開発」と定めた以上、push 頻度は上がる。

### 却下した案: マニフェストでスコープを絞る

CI の 2 検査と同じスコープにフックを揃える案を検討し、**却下した。この機構に穴を開けるため。**

**理由 1: マニフェストは配布面ではない。**

| パス | マニフェスト | 配布先への影響 |
|---|---|---|
| `template/` | 非掲載 | scaffold が新規プロジェクトに丸ごとコピーする |
| `scripts/scaffold.sh` | 非掲載 | 全 consumer が導入時に実行する |
| `docs/` | 非掲載 | 直接は届かないが、kit の挙動を説明する正典 |

`template/` の骨格に特定スタックの前提が混ざれば、以降の新規プロジェクト全部が継承する。`scaffold.sh` に固有名が入れば全 consumer の導入が壊れる。混入の被害範囲としてはむしろ大きい。

**理由 2: 3 つの検査は別の問いに答えている。**

| 検査 | 問い | スコープを絞る理由 |
|---|---|---|
| CI 鮮度 | この写しは最新の kit 由来か | 丸ごとコピーされる経路が無いファイルにはロストアップデートが起きない。トレーラを要求しても bot の自動 PR が無意味に落ちるだけ |
| CI 混入 grep | 追加行に禁止語があるか | 設計文書は経緯として正当にスタック名に言及する。単純一致では区別できず、見ると形骸化する |
| **審査 agent** | **この内容は kit に属するか** | **絞る理由が無い。この問いはマニフェストより広い範囲に及ぶのが正しい** |

CI の 2 検査が絞っているのは「そのファイルにはその検査が原理的に効かない／有害」だからであって、「そのファイルは重要でない」からではない。**「CI と非対称だから揃えるべき」という推論は誤り。**

**理由 3: 非掲載ファイルへの指摘は実際に出ている。**

`docs/` 移設 PR で、審査 agent は `kit-push-gate.md` を contaminated と判定した（doc の記述が `classifyPushCommand` の現挙動と食い違っていた。agent は実際にコードを実行して確認したうえで指摘した）。マニフェストで絞っていれば素通りしていた。**「審査対象がゼロなら必ず clean になる」は成り立たない。**

### 摩擦を下げるなら

目的は「マニフェスト外の変更をまとめて安全扱いする」ことではない。**kit の内容・配布物・挙動の正典に影響しないと個別に説明できる変更だけ**を、毎回の agent 起動から外すことである。審査は差分に対して行うので、全変更パスが明示的に免除されている push だけを免除する。変更パスのうち 1 つでも免除外なら、従来どおり差分全体について review agent を起動する。

つまり、これは「レビュー対象をマニフェストへ縮小する」仕組みではなく、**狭く・理由付きで選んだ no-review の例外**である。`README.md`・`CLAUDE.md` を一括で免除する意図はない。これらは kit の使い方・境界を定める正典だからである。

**`template/` と `scripts/` は必ず審査対象に残すこと。**

免除リストを実装するときの契約:

- パスはリポジトリルート相対で明示する。ディレクトリを免除するなら再帰を意図したことが分かる記法にし、暗黙の prefix 一致にはしない
- リストへの追加・削除そのものは常に審査対象にする。自分を免除してから未審査で変更する経路を作らないため
- 各エントリには「なぜ kit の内容・配布物・正典・ゲートのどれにも影響しないか」という根拠を残す。単に頻繁に変わることは免除理由にならない
- `template/`・`scripts/` に加え、ルートの正典文書、フック・CI・マニフェストは免除しない

#### 採用した免除: `docs/`

ルートの `docs/` は、配布物ではなく workflow-kit 自身の**設計・実装・判断理由・未解決問題の集積所**である。ここだけを変更するたびに review agent を必須にすると、設計記録を残すコストが不釣り合いに高くなる。したがって `docs/` 配下だけの変更は審査免除とし、内容の最終確認は PR の人間レビューに委ねる。

これは `docs/` に実装との不整合を検出した実績があるという理由 3 との意図的なトレードオフである。その検査を失うことを理解したうえで、設計記録の継続的な更新を優先する。`docs/` と他のパスを同時に変更した場合は、免除せず従来どおり差分全体を審査する。

具体的なリストはリポジトリルートの `.kit-push-review-exemptions.json` に置く。このファイル自身の変更は常に審査対象である。

### 追記（2026-07-13）: 免除リストを審査スコープの宣言に置き換えた

上記の免除リスト方式は実装して配布したが、**stack 層の kit（実アプリでもある）に展開した時点で破綻した**。方針を転換したので経緯を残す。

**破綻の内容。** 免除は denylist であり、「配られないもの」を全部数え上げる必要がある。それは kit 単体なら `docs/` だけで済むが、stack 層の kit は同時に実アプリであり、`src/`・テスト・設定ファイルなど**リポジトリの大半が配られない**。数え上げは現実的でない。

結果、stack kit では**アプリのコードを1行直して push するだけで、審査すべき配布物が1つも無いのに verdict を要求された**。直近のコミットを数えると、審査が本当に要るのは9件、要らないのに審査されるのが13件、免除リストが実際に免除したのは0件だった（免除は「変更集合が全部免除パスに収まるとき」にしか効かず、設計記録を書くコミットはたいてい他のファイルも伴うため）。

**転換。** 「配られないもの」を数え上げるのをやめ、**審査スコープを正の方向で宣言する**。審査は `変更集合 ∩ 審査スコープ` が空でないときだけ走る。スコープは以下の和である。

- フックが持つ `MANDATORY_SCOPE` — `README.md`・`CLAUDE.md`（kit の境界を定める正典）、`.github/`・`.githooks/`（ゲート定義）、`template/`・`scripts/`（scaffold で外へ出る経路）、および宣言ファイル自身
- マニフェスト（`*-kit-files.txt`）の配布物 — 配布経路の大半はこれで尽きる
- `.kit-push-review-scope.json` の `review_extra` — その kit に固有の追加パス

これは上の「レビュー対象をマニフェストへ縮小する仕組みではない」という記述を覆すものではない。**マニフェストへの縮小ではない。**「`README.md`・`CLAUDE.md` を一括で免除する意図はない」「`template/`・`scripts/`・フック・CI・マニフェストは免除しない」という当時の契約はすべて維持されている。

#### 実装中に審査 agent が見つけた穴: 宣言忘れが無審査に倒れていた

最初の実装は `MANDATORY_SCOPE` を持たず、スコープを「マニフェスト ∪ 宣言ファイルの `review_extra`」とし、**宣言ファイルが無ければマニフェストのみ**としていた。審査 agent が最小の kit クローン（マニフェストだけ・宣言ファイル無し）を実際に作り、`README.md`・`CLAUDE.md`・`template/` 配下を変更しても `review_paths` が空になることを実証して contaminated と判定した。

これは denylist から allowlist へ反転させたことによる**フェイルオープン**である。免除リスト方式では「宣言し忘れ＝全部審査」で安全側に倒れていたのに、宣言方式では「宣言し忘れ＝無審査」になる。しかもフックは配布されるが宣言ファイルは配布されないため、**この穴は配布先の kit すべてで開く**。

2段構えで塞いだ。

1. **宣言ファイルが無ければ全変更を審査する。** スコープを狭めるのは、狭めてよい範囲を明示的に宣言した kit だけの特典とする。
2. **構造的なスコープをフック側に `MANDATORY_SCOPE` として持つ。** 正典・ゲート定義・scaffold 配布経路は、どの kit にも同じ役割で存在する。個別の kit の宣言漏れでゲートが外れないよう、宣言ファイルではなくフックに持たせる。

この過程で、最初の宣言が `.githooks/`（`pre-push` ゲート）を落としていたことも同時に発見された。`.github/`（CI ゲート）を審査対象としながら同じ役割の `.githooks/` を落とすのは内部矛盾であり、これも `MANDATORY_SCOPE` に入れた。

#### 最終形: 審査するのは外へ出るものだけ

上記の `MANDATORY_SCOPE` は `README.md`・`CLAUDE.md`・`.github/`・`.githooks/` を「配布はされないが正典・ゲート定義だから」という理由で含んでいた。これも外し、**スコープを配布物だけに揃えた**（`template/`・`scripts/`・マニフェスト掲載ファイル、および宣言ファイル自身）。

理由は一貫している。**審査1回はエージェント起動ぶんの金と時間である。** 混入審査が守るのは「配布先で意味をなさない語彙が、外へ出るファイルに混ざらないこと」であり、外へ出ないファイルには混ざる先が無い。正典やゲート定義の誤りは PR の人間レビューと main の保護（ruleset・CI）で受け止める。自動審査を二重にかけるコストに見合わない。

唯一の例外が宣言ファイル自身である。外すと「スコープを空にする変更」が無審査で通り、ゲートを自分で消せる。

以下は、この最終形に至る途中の判断の記録である。

#### `README.md` は `MANDATORY_SCOPE` から外した（先に外した分）

当初 `README.md` を「kit の境界を定める正典」として `MANDATORY_SCOPE` に入れた。審査 agent は実際にそこで虚偽記述（「CI が理由付き分類を強制する」と書いたが、当時の CI の `jq` は `.path` の一致しか見ていなかった）を捕まえており、doc とコードの整合検査としては機能していた。

それでも外す。**理由は、審査1回がエージェント起動ぶんの金と時間だからである。**

そもそも README に設計文書を書いていたことが誤りだった。README はプロジェクトの概要と最小限の用法に留めるものであり、**大袈裟な審査を通さずとも誤りが分かる程度にコンパクトに保つべきもの**である。配布もされない。審査コストを払う先としては割に合わない。

正しい配置は、内容を `docs/`（`kit-push-gate.md`）に置き、README からは1〜2行で参照することである。そうすれば README が実装とずれる余地自体が小さくなり、審査で守る必要がなくなる。審査予算は配布物に集中させる。

`CLAUDE.md` は `MANDATORY_SCOPE` に残す。こちらは Claude への指示書であり、実装とずれると**挙動そのものが変わる**。README とは性質が違う。

**理由 3（非掲載ファイルへの指摘実績）とのトレードオフは受け入れる。** `docs/` は審査対象から外れるため、doc とコードの不整合を agent が捕まえる経路は失われる。これは免除リスト導入時（2026-07-13）に既に受け入れた損失であり、今回新たに失うものではない。

**宣言漏れの検知。** スコープから漏れる新しいパスが生まれる道は2つしかない。

1. 新しいトップレベルのパスが増える → `kit-push-guard` CI が検知し、`review_extra` か `no_review` への分類を強制する（理由必須）
2. 既存の scaffold スクリプトが新しいディレクトリを配り始める → `scripts/` 自体が審査対象なので agent が読む

既存のトップレベルを全部列挙させる方式は採らない。無関係な構造変更のたびに CI が赤くなり、宣言が儀式化するため。**増えたときだけ**分類を強制する。

**digest の範囲も揃えた。** 従来 digest は変更パス全体から計算していたため、審査を受けたあとに配布物でないファイルを1行触るだけで verdict が無効化された。審査していないものを digest に含めない。

### 既知のバグ: ref の削除も verdict を要求される

pre-push フックが **ref の削除も push として扱う**。マージ済みブランチを消そうとすると、

```
$ git push origin --delete <branch>
kit への push には kit-push-review-agent による clean verdict が必要です。
```

で落ちる。削除には内容が無いので verdict は原理的に取りようがなく、**`git push --delete` が誰にも実行できない**。回避策は GitHub の PR 画面の Delete branch ボタン（git push を通らないためフックにかからない）。

修正は、pre-push の stdin から ref を読み、削除（new sha が全ゼロ）のときは verdict を要求せず通す分岐を足すだけ。未着手。

## 判断していないこと

- そもそも移行するか。core への push は「実作業中に気づいた修正のための副次経路」であり、主戦場は kit リポジトリでの直接開発（`CLAUDE.md`）。頻度が低いなら、現行の非対称性を認識したうえで放置する選択もありうる
- 移行するとして、ローカルフックを完全に消すか、早期フィードバック用の advisory として残すか
- 審査の摩擦を下げるか。下げるとして、免除リストをどこまで広げるか
