# scaffold.sh の導入挙動

最終更新: 2026-07-24
ステータス: 実装完了・稼働中（既存プロジェクトへの安全性は実行検証済み）

`scripts/scaffold.sh` が既存プロジェクトに対しても安全に core を導入できることの根拠と、実際に踏んだ検証・エッジケースの記録。README には結論だけを置き、詳細はここに置く。

## 保証している性質

`scaffold.sh` は「配置とレポートのみ」（git 操作をしない）という設計方針を、以下の実装で担保している。

| 性質 | 実装 |
|---|---|
| 既存ファイルを上書きしない | コピー対象ごとに `[ -e "$dst" ]` を見て、存在すれば `cmp` で同一/差分を判定するだけ。書き込むのは存在しないときだけ |
| `.gitignore` の重複追記防止 | `grep -qxF` で行の完全一致を見てから追記する |
| 再実行しても安全（冪等） | 上記2点の結果、同じ入力に対して同じ出力になる。git 操作を一切しないため、再実行によるコミット・push の重複も発生しない |
| 差分を無視して同期を破壊しない | 下記「base.txt の三方向比較ハザード」を参照 |
| hooks を発火させる状態まで導入する | 最後に `node .claude/scripts/merge-hook-registrations.cjs` を自動実行し、`settings.json` に不足分だけ追記する。同スクリプト自身が冪等・想定外構造での書き込み拒否を保証しているため、自動実行に追加のリスクはない。`node` が無い環境では自動登録をスキップし、手動実行を促すレポートを出す |

## base.txt の三方向比較ハザード

`workflow-kit-base.txt`（kit HEAD の SHA）は、pull-check CI が「kit / project / base」の三方向比較で差分の意味を判定するための祖先ポインタである。

**差分あり（`skipped_diff`）が1件でも残っている間は書かない。** 理由:

差分が残ったまま base を kit の現在の HEAD に進めると、三方向比較は「base と kit が同一・project だけが違う」という形になり、その違いを「project 側の意図的な先行変更（local）」と誤分類する。実態は project が kit の内容にまだ追随できていないだけなのに、以後 kit 側の更新は pull で取り込まれなくなり、CI は「kit へ push してください」という**逆向きの notice** を出し続ける。

この誤分類が危険なのは、「要解決」が消えずに残るのではなく「解決済み・project 優先」として**見えなくなる**ことである。差分は `/workflow-kit-pull` または `/workflow-kit-push` で解決してから base を設定する。

## 実行検証ログ（2026-07-24）

読解だけでなく、scratchpad に使い捨ての git リポジトリを作って実際に `scaffold.sh` を実行して確認した。

**セットアップ**（「既存プロジェクト」を模した状態）:

- `.claude/rules/commit-guide.md` … kit と無関係な独自内容で先に存在
- `.claude/rules/grill-me.md` … kit と全く同じ内容で先に存在
- `.gitignore` … `node_modules/` と `.claude/steering/` が既に記載された状態
- `.claude/steering/current.md` … 進行中タスクのメモが既に存在

**確認できた結果:**

| 検証項目 | 結果 |
|---|---|
| 内容が異なる既存ファイル | 上書きせず「差分あり」報告のみ、内容は無傷 |
| 内容が同一の既存ファイル | 「同一」報告、無変更 |
| 既存 `.gitignore` の行 | 重複追記なし。無い行だけ追記 |
| 既存の `steering/current.md` | 無変更で保持 |
| 差分が残っている間の `base.txt` | 書かれない（上記ハザードの回避を実測） |
| 差分を解消せず再実行 | 完全に同じ結果、副作用ゼロ |
| サブディレクトリから実行 | エラーで即停止、書き込みなし |
| kit 自身のルートから実行 | エラーで即停止、書き込みなし |

`git status` でも削除・改変は一切なく、追加のみであることを確認した。

### 発見したエッジケース（低リスク・未対応）

`.claude/hooks` のように kit がディレクトリとして扱うパスに、既存プロジェクトが**同名のファイル**を既に置いていた場合、Step 2 の `mkdir -p` が `File exists` で失敗し、`set -e` によりレポートを出す前に即終了する（ファイル配置ロジックにすら到達しない）。

- データは壊れない（既存ファイルの内容は無傷。実行前後で `cat` の出力が一致することを確認済み）
- 失敗時に何の報告も出ない点は改善の余地があるが、対象パスはすべて `.claude/` 配下という Claude Code の予約領域であり、実プロジェクトが同名の非 kit ファイルを既に持っている可能性は事実上ゼロ。対応の優先度は低いと判断し、未対応のまま記録するに留める

## scaffold-template.sh との違い

上記の検証は **core**（`scaffold.sh`）のみを対象にしている。template 層（`scaffold-template.sh` + `/template-setup`）は別物で、既存プロジェクトへの後付けはまだ実証されていない。

- 実証済みなのは捨てプロジェクト（`kit-smoke-fastapi`）への**新規セットアップ**のみ（[template-scaffold.md](template-scaffold.md) の試運転ログ）
- `hono-user-point` は `hono-auth-starter`（Template repository）から複製された孫プロジェクトであり、新規プロジェクト作成チェーンの実証に含まれる。「既にコードがある既存プロジェクトへ template 層を後付けする」検証ではない
- README の導入手順が `scaffold-template.sh` を「（新規プロジェクトの場合）」と明記しているのはこのため

既存プロジェクトへ template 層を導入したい場合は、本番環境に直接試す前に小さな実験を挟むこと。

## 修正履歴

### hooks 登録（settings.json への反映）を手動手順から自動実行に変更（2026-07-24）

**種別:** 仕様変更（軽微）
**対象ファイル:** `scripts/scaffold.sh`

**変更内容:** 最終レポートの前に `node .claude/scripts/merge-hook-registrations.cjs` を自動実行するよう変更。従来は「残りの手動作業」としてユーザー自身の実行を要求していた。`node` が見つからない場合・登録スクリプトが異常構造を検知して失敗した場合は、レポートにその旨を記録し手動実行を促すことで、実行不能時にも scaffold.sh 自体は正常に完了する。
**理由:** 手動ステップとして残す限り、実行を忘れる・見落とすリスクが常につきまとう。`merge-hook-registrations.cjs` 自身が「不足分だけ追記・冪等・想定外構造では書き込まず exit 1」という安全な契約を既に持っており、自動実行に踏み込んでも安全側の性質は変わらないため、手動運用に留める理由がないと判断した。
